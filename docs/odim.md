# ODIM

## Outcome, Decision, Insight, Measure

ODIM, credited to Larry Maccherone, is a simple way to teach and design better metrics.

It exists to answer a common problem in engineering and product organizations:

**Measures are easy to collect, but much harder to connect to the outcomes the business actually wants.**

## The four parts

### Outcome
What business, customer, product, or strategic result are we trying to achieve?

### Decision
What decision do we need to make to improve that outcome?

### Insight
What understanding would help us make that decision better?

### Measure
What observable signal helps generate that insight?

## The forward direction: ODIM

1. Start with the **Outcome**
2. Identify the **Decision**
3. Clarify the **Insight**
4. Choose the **Measure**

Example:

- **Outcome:** Improve customer retention
- **Decision:** Which workflow improvements should we prioritize?
- **Insight:** Which workflows are most associated with retained customers?
- **Measure:** Adoption and repeat usage of those workflows by retained vs churned accounts

## The reverse direction: MIDO

1. Start with the **Measure**
2. Ask what **Insight** it can support
3. Ask what **Decision** that insight could improve
4. Ask what **Outcome** that decision connects to

If a measure does not support an important insight, decision, or outcome, it may be much less useful than it appears.

## Why teams oscillate between ODIM and MIDO

In real organizations, metric design is rarely one-directional.

Teams often move:
- from outcomes downward
- then from existing measures upward
- then back down again to refine

That is normal.

## ODIM and VISTA

For each VISTA dimension, ODIM helps teams connect measures to outcomes.

### Value
- **Outcome:** Increase retention
- **Decision:** Which feature should be improved first?
- **Insight:** Which feature usage patterns correlate with retention?
- **Measure:** Repeat usage by retained vs churned customers

### Impact
- **Outcome:** Win in a target segment
- **Decision:** Which capability should be prioritized?
- **Insight:** Which capabilities influence buying confidence or activation success?
- **Measure:** Segment-specific capability adoption and downstream conversion

### Safety
- **Outcome:** Reduce customer-facing regressions and shipped security failures
- **Decision:** Which changes need deeper review?
- **Insight:** Which change types create the most post-merge reliability and security problems?
- **Measure:** Regression rate, security issue rate, and cleanup burden by change type

### Timing
- **Outcome:** Improve strategic sequencing
- **Decision:** Which item should go next?
- **Insight:** Which option has the highest cost of delay?
- **Measure:** Capacity allocation to high-cost-of-delay work and hindsight decision quality

### Adaptation
- **Outcome:** Reduce repeated engineering mistakes
- **Decision:** How should similar future work be reviewed or routed?
- **Insight:** Which issue patterns keep causing trouble?
- **Measure:** Regression rate by repeated issue type over time

## Summary

ODIM helps teams move beyond counting visible activity and toward designing measures that create insight, improve decisions, and contribute to real outcomes.

# Value

## Definition

**Value** asks a simple but important question:

**Did this work create meaningful user or business benefit?**

In VISTA, value is not measured by code written, pull requests merged, tickets closed, or features released. Those are outputs. Value is about whether the work made a meaningful difference for customers, the business, or both.

As AI makes software output cheaper and faster to produce, value becomes more important, not less. When teams can build more, the risk is not that they build too little. The risk is that they build more of the wrong things, or that they mistake shipped work for realized benefit.

Value is the first guard against that failure mode.

---

## Why it matters

Modern engineering teams have no shortage of activity metrics. They can usually tell you:

- how many pull requests were merged
- how many story points were completed
- how fast code reviews moved
- how often software was deployed

What they often cannot tell you is whether any of that mattered.

This is one of the core problems VISTA is designed to address.

Without an explicit value lens, organizations drift toward:

- delivery theater
- feature accumulation
- local optimization
- AI-generated slop
- weak product focus
- backlog clearing mistaken for progress

Value helps reconnect engineering effort to the reason the effort exists in the first place.

---

## What Value is not

Value is **not**:

- output volume
- engineering busyness
- implementation elegance alone
- how much code AI generated
- how quickly something shipped
- how loudly a stakeholder wanted it

Those may matter in context, but they are not evidence of value by themselves.

A feature can ship quickly and still have weak value.
A pull request can be elegant and still solve a low-priority problem.
An AI-generated implementation can save time and still create no meaningful business outcome.

---

## What Value includes

Value should be evaluated in terms of real user and business benefit.

In practice, this often includes:

- adoption by intended users
- repeat usage over time
- measurable movement in a target KPI
- customer trust or satisfaction improvement
- reduction in customer pain or friction
- better completion of an important customer task
- revenue contribution or revenue enablement
- retention, expansion, or churn reduction
- progress against a strategic product objective

In VISTA, value is usually strongest when it is not just visible at launch, but continues to show up in actual use.

That is why one-time clicks or initial curiosity are not enough. Value should be grounded in meaningful traction.

---

## The core idea

A useful shorthand is:

**Value is realized benefit, not delivered output.**

That means the work should be judged not by whether it was implemented, but by whether it changed something important for the better.

---

## Typical signals of strong Value

Indicators that a work item or stream is creating value include:

- intended users actually adopt it
- those users continue to use it over time
- the change improves an important user or business metric
- customer friction is reduced in practice
- the feature or workflow becomes part of normal behavior
- the benefit is large enough to matter relative to alternatives
- the outcome aligns with an explicit business or product goal

A strong value signal is usually a combination of:
- adoption
- repeat usage
- business or strategic movement

---

## Typical signals of weak Value

Indicators of weak value include:

- a feature ships but usage remains low
- adoption exists, but repeat usage does not
- work is technically complete but creates little real-world traction
- the feature is hard to activate or operationalize
- users are confused about what it is for
- the value case depends mostly on internal enthusiasm
- teams celebrate release activity without evidence of customer benefit

In a healthy system, weak value should not be treated as a minor detail. It should trigger reflection about prioritization, customer understanding, and whether the work should have been done in the first place.

---

## Value and AI engineering

AI increases the urgency of value thinking.

Why?

Because AI lowers the cost of producing outputs:
- more code
- more pull requests
- more experiments
- more backlog clearing
- more features

That can be helpful. But it also means organizations can create more low-value work faster than ever before.

In the AI era, value is not a “nice to have” metric. It is a control mechanism against overproduction and misdirected effort.

A team that can produce 3x more output without increasing value has not necessarily become 3x more productive.

It may simply have become better at producing activity.

---

## Common anti-patterns

### 1. Shipping without traction
A feature is considered successful because it launched, even though adoption remains weak.

### 2. One-time novelty mistaken for value
A feature gets initial clicks, but not sustained usage.

### 3. Internal convenience framed as customer value
The benefit is mostly for the team, but is described as if it solves a major user problem.

### 4. Backlog clearing mistaken for product progress
Work gets completed because it was available, not because it was valuable.

### 5. AI-generated volume mistaken for impact
The organization confuses increased implementation speed with increased benefit.

---

## Questions Value should force

Before work starts:
- What user or business benefit should this create?
- For whom?
- How will we know if it worked?
- What would meaningful adoption look like?
- What would repeat usage look like?
- What is the real problem this solves?

After work ships:
- Did the intended users adopt it?
- Did they keep using it?
- Did it improve the metric or outcome we expected?
- Was the value large enough to matter?
- Did the value justify the total cost?

---

## Suggested measures

Teams should choose measures that fit their context, but common examples include:

- adoption rate by intended user group
- repeat usage or retention of feature usage
- usage depth in meaningful workflows
- task completion rate
- customer satisfaction or trust indicators
- movement in KPI tied to the work
- support burden reduction
- revenue lift or revenue enablement
- churn reduction
- percentage of delivered work with meaningful traction

The exact measure matters less than the discipline: the metric should help show whether the work created meaningful benefit.

---

## ODIM example for Value

One useful way to think about Value is through **ODIM**:

- **Outcome:** Increase customer retention
- **Decision:** Which workflows or features should we improve first?
- **Insight:** Which behaviors or capabilities are associated with stronger retention?
- **Measure:** Adoption and repeat usage of those workflows by retained vs churned customers

This helps make sure the measure is not floating on its own. It is tied to a real business outcome and the decisions people need to make.

---

## What good looks like

A team with a strong Value discipline:

- defines intended outcomes before coding starts
- measures adoption and repeat usage, not just release
- distinguishes curiosity from meaningful traction
- stops celebrating output without benefit
- learns which types of work reliably produce real value
- uses value evidence to shape future prioritization

---

## Summary

Value is the dimension that asks whether the work actually mattered.

It is the first correction against output obsession, feature bloat, and AI-driven overproduction. It helps teams see whether engineering is creating real user and business benefit, not just generating more activity.

**VISTA uses Value to make sure software delivery is connected to meaningful outcomes, not just visible output.**
